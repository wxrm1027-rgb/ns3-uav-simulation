# 异构 NMS 仿真平台 — 框架架构与可视化说明

本文档整理自对 `scratch/heterogeneous-nms/framework/heterogeneous-nms-framework.cc` 与 `export_visualization_data.py`、`visualization/` 的问答与代码分析，涵盖：**框架整体架构与核心模块**、**可视化实现方式与内容**、**数据采集 / 存储 / 展示**。

---

## 一、整体框架架构

### 1.1 设计目标与入口

- **设计目标**：构建“分层异构”网络管理仿真底层框架，便于算法接入；清晰分离**拓扑构建**（LTE / WiFi Ad-hoc / 战术数据链）、**应用逻辑**（自定义 Application）与**仿真控制**；预留智能调度/流控/管理算法入口。
- **入口**：`main()` 解析命令行 → 创建 `HeterogeneousNmsFramework` → 调用 `Run(simTime)`。
- **核心调度**：`Run()` 依次执行：建目录与日志 → 构建拓扑（GMC → Adhoc → DataLink → LTE）→ 补全移动模型 → SPN 选举 → 回程 → 静态路由 → 安装应用 → 周期性 JSONL / 业务流日志 → 监控(FlowMonitor) → `Simulator::Run()` → 后处理(FlowMonitor 导出、性能统计、可选 pcap 解析)。

### 1.2 核心模块清单

| 序号 | 模块名称 | 代码位置（约） | 类型 |
|------|----------|----------------|------|
| 1 | 统一日志 NmsLog / JSONL | 49–195 行 | 全局函数 + 全局流 |
| 2 | 时序入网配置 NodeJoinConfig / LoadNodeJoinConfig | 82–195 行 | 结构体 + 全局函数 |
| 3 | TLV 包格式 NmsTlv | 196–261 行 | 命名空间 |
| 4 | 包解析 NmsPacketParse | 262–384 行 | 命名空间 |
| 5 | 节点应用 HeterogeneousNodeApp | 385–934 行 | Application 子类 |
| 6 | GMC 策略应用 GmcPolicySenderApp | 935–1137 行 | Application 子类 |
| 7 | 仿真框架 HeterogeneousNmsFramework | 1139–2284 行 | 框架类 |
| 8 | main 与 CommandLine | 2430–2477 行 | 程序入口 |

各模块的设计目标、核心功能、输入输出及在仿真流程中的作用见下文“模块说明”；模块间通过 `Run()` 调用、NmsLog/JSONL 输出、应用安装与 NmsTlv 组包等形成交互。

### 1.3 模块说明（摘要）

- **NmsLog / JSONL**：统一写文本日志（`log/nms-framework-log_*.txt`）和 JSONL 状态日志（`log/nms-state_*.jsonl`）；为事件回放与自研可视化提供数据源。
- **NodeJoinConfig / LoadNodeJoinConfig**：从 JSON 解析节点入网配置（node_id、join_time、init_pos、speed、type、subnet），驱动时序入网与可视化中的“未入网/已入网”与链路显示。
- **NmsTlv**：统一 TLV 包格式（Type 0x01–0x04），LTE/Adhoc/DataLink 差异化上报与策略包格式。
- **NmsPacketParse**：可选运行时包解析（TLV/Hello/Policy），写 NmsLog，便于调试与离线分析。
- **HeterogeneousNodeApp**：承载节点状态上报、SPN 聚合、策略接收、Hello 选举；TLV 组包与 NmsTlv 一致；算法插入点在 SendPacket/UpdateUvMib/CalculateUtilityScore。
- **GmcPolicySenderApp**：GMC 周期向各 SPN 发送策略包（TLV 0x04）。
- **HeterogeneousNmsFramework**：编排拓扑构建、回程、路由、应用安装、监控、归档；提供 GetNodeJoinTime、WriteAllNodesJsonl 等供时序入网与 JSONL 输出。
- **main**：解析 simTime、topology、parsePackets、pcap、dualChannel、joinConfig 等，注入框架并启动仿真。

## 二、可视化实现方式（自研 Web 可视化）

**您所指的可视化** = 由 **`export_visualization_data.py`** 导出数据 + **`visualization/`** 前端（index.html、app.js、styles.css）加载并展示的整套链路，而非 NetAnim 查看器。

### 2.1 技术方式

- **导出层**：Python 脚本 `export_visualization_data.py`，从仿真结果目录读取 log、JSONL、FlowMonitor XML，解析并生成前端所需的 JSON。
- **展示层**：静态 Web（HTML + JavaScript + CSS），使用 ECharts、SVG 绘制拓扑与图表；通过本地 HTTP 服务（如 `python3 -m http.server`）加载同源下的 JSON 文件。
- **数据流**：仿真结束 → 结果目录 → 运行导出脚本 → 得到 `topology.json`、`events.json`、`stats.json` → 前端加载这三个文件并渲染。

### 2.2 具体实现的可视化内容

| 内容 | 实现方式 |
|------|----------|
| **节点状态** | 角色/子网、入网时间(joinTime)、位置随时间(positions/nodeStatesByTime)、IP/能量/链路质量；未入网灰色、入网后按子网着色 |
| **网络拓扑** | 节点 + 链路(nodes/links)；链路按 link.t 与两端 joinTime 仅在两端入网后显示；去重连线；回程红粗线、数据蓝虚线 |
| **事件时间线** | events.json 事件列表 + 底部时间轴；播放/暂停/重置；点击事件跳转时刻 |
| **业务流高亮** | stats.businessFlows（ROUTE_START/ROUTE_END 解析）；在 startTime～endTime 内按 path 在拓扑上画高亮路径，可勾选显示 |
| **性能指标** | stats.flows（FlowMonitor 解析）、stats.timeSeries、stats.spnElection*；趋势/性能/SPN 选举图表，可导出 PNG/CSV |
| **节点详情** | 点击节点弹出悬浮面板：Node ID、IP、类型、子网、入网状态、能量、链路质量、速度、Score、承载业务、发包/收包；数据来自 topology 节点属性 + stats.nodeStatesByTime 当前时刻 |

## 三、可视化数据的采集、存储与展示形式

### 3.1 采集（导出脚本读什么）

脚本在 `run(dir_path, out_dir)` 中从仿真结果目录 `dir_path` 下读取：

| 数据用途 | 来源路径 | 说明 |
|----------|----------|------|
| 事件列表 | `log/nms-framework-log*.txt` 或任意 `.txt` | NmsLog 文本日志 |
| 节点时序状态 | `log/*.jsonl`（如 nms-state_*.jsonl） | NS-3 的 WriteJsonlStateLine 输出 |
| 拓扑与节点位置 | `log/*.jsonl` + `join_config.json` | 节点时序状态与配置驱动 |
| 流性能统计 | `performance/flowmon_stats*.xml` | FlowMonitor 的 XML |

### 3.2 处理与输出（脚本写什么）

- **事件**：`load_events_from_log` 解析 NmsLog 行 → `events`；推断 role/subnet、join_time；输出 **events.json**。
- **节点与拓扑**：从事件与配置得到节点；通过 JSONL 中 pos_x/pos_y 生成节点 **positions** 并覆盖，并补 ip/energy/link_quality；输出 **topology.json**（nodes + links）。
- **节点状态时序**：`load_jsonl_log` + `build_node_states_from_jsonl` → 按时间桶的 ip/energy/link_quality/speed/x/y；输出到 **stats.json** 的 **nodeStatesByTime**。
- **业务流**：`extract_business_flows_from_events` 从事件中解析 ROUTE_START/ROUTE_END → flowId、type、path、startTime、endTime；输出到 **stats.json** 的 **businessFlows**。
- **性能与统计**：FlowMonitor XML → stats.flows；事件中提取 PERF/SPN_ELECT 等 → stats.timeSeries、spnElectionCount、spnElectionBySubnet；一并写入 **stats.json**。

输出目录通常为 `-o visualization/data`，即 **topology.json**、**events.json**、**stats.json**（及可选 meta.json）落在同一目录，供前端加载。

### 3.3 展示形式（前端如何用）

- **拓扑图**：读取 topology.json 的 nodes、links；按当前时间与 joinTime、link.t 决定节点颜色与链路显示；用 nodes[].positions 或 nodeStatesByTime 的 x/y 做节点移动。
- **时间轴**：用 events.json 与当前时间更新“当前时刻”、事件列表高亮；点击事件跳转时刻。
- **节点详情**：用 topology 中节点属性 + stats.nodeStatesByTime 中当前时刻的状态展示 IP、能量、链路质量等。
- **业务流**：用 stats.businessFlows 在拓扑上在 startTime～endTime 内按 path 绘制高亮路径。
- **统计图表**：用 stats.flows、timeSeries、spnElection* 绘制性能、趋势、SPN 选举等图表。

## 四、与 NS-3 仿真的衔接

- **仿真侧**：`scratch/heterogeneous-nms/framework/heterogeneous-nms-framework.cc` 通过 NmsLog 写入 `log/nms-framework-log_*.txt`，通过 WriteJsonlStateLine/WriteAllNodesJsonl 写入 `log/nms-state_*.jsonl`；FlowMonitor 输出 `performance/flowmon_stats_*.xml`。时序入网、业务路由（ROUTE_START/ROUTE_END）等均通过 NmsLog 或 JSONL 体现。
- **导出侧**：`export_visualization_data.py` 仅读取上述文件，不修改 NS-3；运行命令示例：  
  `python3 export_visualization_data.py simulation_results/<时间戳> -o visualization/data`
- **前端**：通过 HTTP 服务打开 `visualization/index.html`，加载 `data/topology.json`、`data/events.json`、`data/stats.json` 即可获得完整可视化。

---

## 五、小结

- **框架**：由 HeterogeneousNmsFramework 统一编排拓扑、应用、监控与归档；核心模块包括日志/JSONL、TLV、包解析、HeterogeneousNodeApp、GmcPolicySenderApp 等，模块间通过 Run()、NmsLog、应用安装与 NmsTlv 衔接。
- **可视化**：以 **export_visualization_data.py + visualization/** 为主；采集仿真结果目录中的 log、visualization、performance；输出 topology.json、events.json、stats.json；前端据此展示拓扑、时序入网、事件、业务流高亮与性能统计，并统一将**后续相关说明文档**放在 **docs/heterogeneous-nms/** 目录下维护。
